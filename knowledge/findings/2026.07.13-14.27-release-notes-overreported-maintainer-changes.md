The v0.6.1 release-note generation treated repository-maintenance changes as
product features: it listed knowledge-base and commit-message guidelines,
contributing documentation, project metadata, and counts of modified/added
files. These items were not shown as changes that users could observe or use.

This matters because a diff-based release-note prompt needs an explicit
user-impact gate and a hard exclusion for CI, internal process, contributor
material, and implementation-only changes. A no-product-impact release should
instead receive one concise internal, documentation-update, or maintenance
classification, and the generated body should not add its own title heading.

Caveat: this finding is based on one generated draft release and the local
skill/workflow revision made afterward; it does not establish that every
release range will produce the same pollution.
